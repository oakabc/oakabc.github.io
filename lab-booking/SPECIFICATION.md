| | |
|---|---|
| **เวอร์ชันเอกสาร** | 1.0 (15 ก.ค. 2569) |
| **เวอร์ชันโค้ด backend** | `CODE_VERSION = 4` |
| **สถานะ** | ใช้งานจริง (Production) |
| **URL ระบบ** | https://oakabc.github.io/lab-booking/ |

## 1. ภาพรวมและวัตถุประสงค์

ระบบจองเครื่องคอมพิวเตอร์ในห้องปฏิบัติการ 56 เครื่อง สำหรับนักศึกษาใช้งานผ่านมือถือ
โดยสแกน QR code ที่ติดหน้าห้อง เปิดให้จองเฉพาะ**วันศุกร์ เวลา 13:00–17:00 น.**
ขั้นต่ำครั้งละ 30 นาที ออกแบบให้ **ไม่มีค่าใช้จ่ายในการรันระบบ** และผู้ดูแลเห็น/แก้ไข
ข้อมูลได้เองผ่าน Google Sheets

**ผู้ใช้งาน 2 กลุ่ม:**
- **นักศึกษา** — ดูผังที่นั่ง จองเครื่อง ยกเลิกการจองของตนเอง
- **ผู้ดูแล (admin)** — ปิด/เปิดเครื่อง (เช่น ส่งซ่อม), ดูรายการจองชื่อเต็ม, ยกเลิกการจองใดๆ

## 2. สถาปัตยกรรม

```
นักศึกษา (มือถือ สแกน QR)
   │ HTTPS
   ▼
GitHub Pages ─────────── โฮสต์หน้าเว็บ static (repo: oakabc/oakabc.github.io, โฟลเดอร์ /lab-booking)
   │ fetch (GET / POST text-plain)
   ▼
Google Apps Script Web App ── REST-like API (ไฟล์ Code.gs, Execute as: เจ้าของ, Access: Anyone)
   │ SpreadsheetApp
   ▼
Google Sheets ────────── ฐานข้อมูล (ชีต Bookings, Blocked)
```

**เหตุผลการเลือก:** ฟรีทั้งหมด, ไม่ต้องดูแล server, admin เปิด Sheet ดูข้อมูลดิบได้ทันที
**ทางเลือกสำรองในโปรเจกต์:** เวอร์ชัน Node.js/Express (`server.js` + `public/`) สำหรับรันในเครื่อง/แลนภายใน — ฟีเจอร์เท่ากันทุกข้อ เก็บข้อมูลใน `bookings.json`

### โครงสร้างไฟล์

| ไฟล์ | หน้าที่ |
|---|---|
| `docs/index.html` | หน้าจองสำหรับนักศึกษา (เวอร์ชัน production) |
| `docs/admin.html` | หน้าผู้ดูแล |
| `docs/qr.html` | โปสเตอร์ QR สำหรับพิมพ์ติดหน้าห้อง |
| `docs/config.js` | จุดตั้งค่า URL ของ API (จุดเดียว) |
| `docs/qrcode.min.js` | ไลบรารีสร้าง QR (bundle ฝังในโปรเจกต์ ไม่พึ่ง CDN) |
| `apps-script/Code.gs` | Backend ทั้งหมด (วางใน Apps Script editor) |
| `server.js`, `public/` | เวอร์ชัน Node สำรอง |
| `DEPLOY.md` | ขั้นตอน deploy ละเอียด |

## 3. ผังห้องและการนับหมายเลขเครื่อง

- เครื่องทั้งหมด **56 เครื่อง** หมายเลข 1–56 นับจากหน้าห้อง ซ้ายไปขวา
- **8 แถว × แถวละ 7 เครื่อง** เว้นทางเดินกลางหลังเครื่องที่ 3 ของแต่ละแถว:

```
        หน้าห้อง / จอโปรเจกเตอร์
[ 1  2  3]  ทางเดิน  [ 4  5  6  7]
[ 8  9 10]  ทางเดิน  [11 12 13 14]
[15 16 17]  ทางเดิน  [18 19 20 21]
...                       ...
[50 51 52]  ทางเดิน  [53 54 55 56]
```

ค่าคงที่: `TOTAL_SEATS=56`, `SEATS_PER_ROW=7`, `AISLE_AFTER=3`
(กำหนดใน `Code.gs` และซ้ำใน `localConfig()` ของหน้าเว็บ — แก้ต้องแก้ทั้งสองที่)

## 4. กติกาการจอง (Business Rules)

| # | กติกา | ตรวจที่ |
|---|---|---|
| BR-1 | จองได้เฉพาะ**วันศุกร์**เท่านั้น | server + UI |
| BR-2 | ช่วงเวลาเปิดจอง **13:00–17:00 น.** แบ่งเป็น 8 ช่อง (slot) ช่องละ **30 นาที** | server + UI |
| BR-3 | จองขั้นต่ำ 1 ช่อง (30 นาที) ต่อเนื่องได้สูงสุดถึง 17:00 (4 ชม.) | server + UI |
| BR-4 | เปิดจองล่วงหน้า **4 วันศุกร์** (รวมวันนี้ถ้าเป็นศุกร์และยังไม่เลย 17:00) | server + UI |
| BR-5 | ห้ามจองวันที่ผ่านมาแล้ว และห้ามจองช่วงเวลาที่ผ่านไปแล้วของวันนี้ | server + UI |
| BR-6 | **ห้ามจองซ้อนทับ** ต่อเครื่องต่อช่วงเวลา (เช็คแบบ overlap: `startA < endB && endA > startB`) ป้องกัน race ด้วย `LockService` | server |
| BR-7 | เครื่องที่ถูก**ปิดให้บริการ** (blocked) จองไม่ได้ จนกว่า admin จะเปิดกลับ | server + UI |
| BR-8 | ชื่อผู้จองอย่างน้อย 2 ตัวอักษร | server + UI |
| BR-9 | การจองสำเร็จได้**รหัสยกเลิก 6 หลัก** — ใช้ยกเลิกรายการนั้นได้โดยไม่ต้องล็อกอิน | server |
| BR-10 | การปิดเครื่อง (block) **ไม่ลบ**การจองที่มีอยู่ — admin ตัดสินใจยกเลิกเอง | server |

**นิยาม slot:** slot `i` (0–7) = เวลา `13:00 + 30×i` นาที เช่น slot 0 = 13:00–13:30, slot 7 = 16:30–17:00
การจองเก็บเป็น `startSlot` (รวม) ถึง `endSlot` (ไม่รวม) เช่น 14:00–16:00 = startSlot 2, endSlot 6

## 5. โมเดลข้อมูล (Google Sheets)

### ชีต `Bookings` — การจอง (1 แถว = 1 รายการจอง)

| คอลัมน์ | ชนิด | ตัวอย่าง | หมายเหตุ |
|---|---|---|---|
| id | UUID string | `48ff0792-...` | สร้างโดย `Utilities.getUuid()` |
| code | string 6 หลัก | `403145` | รหัสยกเลิก — **ห้ามเปิดเผยผ่าน API อ่านทั่วไป** |
| date | string `yyyy-MM-dd` | `2026-07-17` | คอลัมน์ตั้ง format `@` (text); ฝั่งอ่านมี `normDate_()` กันกรณี Sheets แปลงเป็น Date |
| seat | number 1–56 | `12` | |
| startSlot | number 0–7 | `2` | |
| endSlot | number 1–8 | `6` | exclusive |
| name | string | `สมชาย 65010012` | ชื่อเต็ม — แสดงต่อสาธารณะแบบ mask |
| createdAt | ISO string | `2026-07-15T04:11:13Z` | |

### ชีต `Blocked` — เครื่องที่ปิดให้บริการ

| คอลัมน์ | ชนิด | ตัวอย่าง |
|---|---|---|
| seat | number | `7` |
| reason | string | `ส่งซ่อม เมนบอร์ดเสีย` |
| blockedAt | ISO string | `2026-07-15T04:37:22Z` |

ทั้งสองชีตถูกสร้างอัตโนมัติเมื่อถูกเรียกครั้งแรก การลบแถวใน Sheet = ลบรายการนั้นจากระบบ
(ควรทำผ่าน UI แทนการแก้ Sheet ตรงถ้าเป็นไปได้)

## 6. API Specification

Base URL: `https://script.google.com/macros/s/<DEPLOYMENT_ID>/exec`

**ข้อกำหนดการเรียก:**
- ทุก response เป็น JSON และคืน **HTTP 200 เสมอ** (ข้อจำกัดของ Apps Script) — ตรวจ error จาก field `error` ใน body
- POST ต้องส่ง body เป็น JSON string โดย **ไม่ตั้ง header `Content-Type: application/json`**
  (Apps Script ไม่รองรับ CORS preflight — ปล่อยเป็น `text/plain` ที่เบราว์เซอร์ตั้งให้)
- POST ตอบผ่าน redirect 302 → เบราว์เซอร์ตามอัตโนมัติ (curl ต้องใช้ GET ตาม redirect เอง)

### 6.1 GET `?action=config`
ข้อมูลค่าคงที่ + วันศุกร์ที่เปิดจอง ใช้ตรวจเวอร์ชัน deployment
```json
{ "version": 4, "totalSeats": 56, "seatsPerRow": 7, "aisleAfter": 3,
  "totalSlots": 8, "slotMinutes": 30,
  "slotLabels": ["13:00","13:30","...","17:00"],
  "fridays": ["2026-07-17","2026-07-24","2026-07-31","2026-08-07"],
  "debug": { "..." : "ข้อมูลวินิจฉัย (ชั่วคราว)" } }
```
> หมายเหตุ: หน้าเว็บปัจจุบัน**ไม่เรียก** endpoint นี้แล้ว (คำนวณ config ในเครื่องเพื่อความเร็ว) — คงไว้เพื่อการตรวจสอบระบบ

### 6.2 GET `?action=bookings&date=yyyy-MM-dd`
รายการจอง (ชื่อถูก mask) + เครื่องที่ปิด ของวันนั้น
```json
{ "bookings": [ { "id":"...", "date":"2026-07-17", "seat":12,
                  "startSlot":2, "endSlot":6, "name":"สมช***" } ],
  "blocked":  [ { "seat":7, "reason":"ส่งซ่อม" } ] }
```

### 6.3 POST `{action:"book", date, seat, startSlot, endSlot, name}`
สร้างการจอง — ตรวจ BR-1 ถึง BR-8 ทั้งหมดฝั่ง server ภายใต้ `LockService`
```json
// สำเร็จ
{ "ok": true, "booking": { "id":"...", "code":"403145", "date":"2026-07-17",
                           "seat":12, "start":"14:00", "end":"16:00", "name":"สมชาย" } }
// ล้มเหลว (ตัวอย่าง)
{ "error": "เครื่อง 12 ถูกจองแล้วช่วง 14:00–16:00" }
```

### 6.4 POST `{action:"cancel", id, code}`
ยกเลิกการจองด้วยรหัส 6 หลักของผู้จอง → `{ "ok": true }` หรือ `{ "error": "รหัสยกเลิกไม่ถูกต้อง" }`

### 6.5 Admin actions (ทุกคำสั่งต้องแนบ `pin` ตรงกับ `ADMIN_PIN`)

| Action | Body เพิ่มเติม | ผล |
|---|---|---|
| `adminlist` | `date` | รายการจองของวัน (**ชื่อเต็ม**, ไม่มี code) + blocked ทั้งหมด |
| `block` | `seat`, `reason` | ปิดเครื่อง (ปิดซ้ำ = อัปเดตเหตุผล) |
| `unblock` | `seat` | เปิดเครื่องกลับ |
| `admincancel` | `id` | ยกเลิกการจองใดๆ โดยไม่ต้องใช้ code |

PIN ผิด → `{ "error": "รหัสผู้ดูแลไม่ถูกต้อง" }`

### ข้อความ error ทั้งหมดของ `book`
`รูปแบบวันที่ไม่ถูกต้อง` · `จองได้เฉพาะวันศุกร์เท่านั้น` · `ไม่สามารถจองวันที่ผ่านมาแล้วได้` ·
`หมายเลขเครื่องไม่ถูกต้อง` · `ช่วงเวลาไม่ถูกต้อง (13:00–17:00 ขั้นต่ำ 30 นาที)` ·
`ช่วงเวลานี้ผ่านไปแล้ว` · `กรุณากรอกชื่อผู้จอง (อย่างน้อย 2 ตัวอักษร)` ·
`เครื่อง N ปิดให้บริการชั่วคราว (เหตุผล)` · `เครื่อง N ถูกจองแล้วช่วง HH:MM–HH:MM` ·
`ระบบกำลังยุ่ง กรุณาลองใหม่อีกครั้ง` (ขอ lock ไม่ได้ใน 10 วิ)

## 7. หน้าจอ (UI Specification)

### 7.1 หน้าจอง `index.html` (mobile-first, ภาษาไทย)

**Flow:** สแกน QR → เห็นผังทันที → เลือกวันศุกร์ → แตะเครื่อง → เลือกช่วงเวลา → กรอกชื่อ → ยืนยัน → ได้รหัส 6 หลัก

- **เลือกวัน:** ชิป 4 วันศุกร์ (แสดง พ.ศ. เช่น "17 ก.ค. 2569")
- **ผังที่นั่ง:** ปุ่ม 56 ปุ่มตามผังจริง สีบอกสถานะ:
  - 🟩 เขียว `#16a34a` — ว่างทั้งวัน
  - 🟧 ส้ม `#f59e0b` — ว่างบางช่วง
  - 🟥 แดง `#dc2626` — เต็มทั้งวัน
  - ⬛ เทาเข้ม `#334155` + 🔧 — ปิดให้บริการ (แตะแล้วขึ้นเหตุผล จองไม่ได้)
- **เลือกช่วงเวลา:** ปุ่ม 8 ช่อง แตะช่องแรก→ช่องสุดท้ายเพื่อเลือกช่วงต่อเนื่อง (ขยายไม่ได้ถ้าคร่อมช่องที่ถูกจอง), ช่องที่จองแล้วขีดฆ่า, ช่องที่เลยเวลาแล้วเป็นสีจาง
- **รายการจองของเครื่อง:** เวลา + ชื่อ (mask) + ปุ่ม **ยกเลิก** (กรอกรหัส 6 หลัก)
- **ผลการจอง:** การ์ดเขียวแสดงรหัส 6 หลักตัวใหญ่ + คำเตือนให้เก็บรหัส
- **Deep link:** `/?seat=N` เปิดแผงจองเครื่องนั้นทันที (เผื่อติด QR รายเครื่องในอนาคต)
- **พฤติกรรมความเร็ว:** ผัง render ทันทีจาก config ในเครื่อง, มีป้าย "⟳ กำลังอัปเดต…" ระหว่างโหลดข้อมูลจริง, หลังจอง/ยกเลิกอัปเดตผังทันที (optimistic) แล้วซิงก์เบื้องหลัง

### 7.2 หน้าผู้ดูแล `admin.html`

- ล็อกอินด้วย PIN (เก็บใน `sessionStorage` — ปิดแท็บแล้วต้องกรอกใหม่), ปุ่มออกจากระบบ
- ผังที่นั่ง: แตะเครื่องปกติ → กรอกเหตุผล → ปิดเครื่อง / แตะเครื่องที่ปิด → ยืนยัน → เปิดกลับ
- รายการจองรายวัน (ชื่อเต็ม) + ปุ่มยกเลิกรายรายการ
- ธีมสีม่วง `#7c3aed` แยกจากหน้านักศึกษา (สีน้ำเงิน `#2563eb`)

### 7.3 โปสเตอร์ QR `qr.html`

- โปสเตอร์แผ่นเดียว: ชื่อระบบ, QR ขนาดใหญ่ชี้ไปหน้าจอง, ขั้นตอน 1-2-3, กติกาย่อ
- QR สร้างฝั่ง browser และ**ฝัง URL ตามโดเมนที่เปิดหน้าโดยอัตโนมัติ** — ต้องเปิดจากโดเมนจริงก่อนพิมพ์
- รองรับ `Ctrl/Cmd+P` (ซ่อน toolbar ตอนพิมพ์)

## 8. ความปลอดภัยและความเป็นส่วนตัว

| ประเด็น | การจัดการ |
|---|---|
| ชื่อผู้จองต่อสาธารณะ | mask เหลือ 3 ตัวแรก + `***` (ชื่อเต็มเห็นเฉพาะ admin และใน Sheet) |
| สิทธิ์ยกเลิก | ต้องมีรหัส 6 หลักของรายการนั้น หรือ PIN ของ admin |
| Admin auth | PIN เดี่ยว (`ADMIN_PIN` ใน Code.gs) — URL หน้า admin เป็นสาธารณะ ความปลอดภัยอยู่ที่ PIN เท่านั้น **ต้องเปลี่ยนจากค่าเริ่มต้น `000000` ก่อนใช้จริง** |
| Race condition | `LockService.getScriptLock()` ครอบทุกการเขียน |
| Rate limiting | ไม่มี (พึ่ง quota ของ Google) — ความเสี่ยงระดับห้องเรียน ยอมรับได้ |
| ข้อมูลใน transit | HTTPS ทั้งเส้นทาง |

**ข้อจำกัดที่ทราบ:** ไม่มีการยืนยันตัวตนนักศึกษา (ใครกรอกชื่ออะไรก็ได้) — ถ้าต้องการบังคับ
ต้องเพิ่ม login (เช่น Google OAuth จำกัดโดเมนมหาวิทยาลัย) ซึ่งอยู่นอกขอบเขตปัจจุบัน

## 9. Performance และขีดจำกัด

| หัวข้อ | ค่า |
|---|---|
| เวลาตอบของ Apps Script | ~1.5–4 วินาที/คำสั่ง (ข้อจำกัดของแพลตฟอร์มฟรี) |
| การโหลดหน้าแรก | ผังแสดงทันที (<100ms), ข้อมูลจองตามมาเบื้องหลัง 1 call |
| Quota ฟรีของ Apps Script | ~20,000 URL fetch executions/วัน — เกินพอสำหรับห้องเดียว |
| ขนาดข้อมูล | เชิงเส้นตามจำนวนแถว อ่านทั้งชีตทุกครั้ง — เหมาะถึงหลักพันแถว ควรลบ/archive แถวเก่าปีละครั้ง |
| อัปเกรดถ้าต้องเร็วกว่านี้ | ย้าย backend ไป Cloudflare Workers (ฟรี) หรือ Render (~$7/เดือน) — หน้าเว็บแก้แค่ `config.js` และรูปแบบ endpoint |

## 10. การ Deploy และการดูแล

### สภาพแวดล้อม production ปัจจุบัน

- **Frontend:** repo `oakabc/oakabc.github.io` โฟลเดอร์ `lab-booking/` → push แล้ว GitHub Pages อัปเดตเองใน ~1 นาที
- **Backend:** Apps Script ผูกกับ Google Sheet ของเจ้าของระบบ, Deploy เป็น Web app (Execute as: Me / Access: Anyone), Timezone: Asia/Bangkok

### ขั้นตอนแก้โค้ด backend (สำคัญ — พลาดบ่อย)

1. แก้/วางโค้ดใน Apps Script editor
2. เพิ่มค่า `CODE_VERSION` ทุกครั้งที่แก้
3. **Deploy → Manage deployments → ✏️ Edit → Version: New version → Deploy**
   (ห้ามใช้ "New deployment" — URL จะเปลี่ยนแล้วหน้าเว็บพัง)
4. ตรวจว่าโค้ดใหม่ทำงาน: เปิด `<URL>?action=config` เช็คค่า `"version"`

### กติกาการดูแลข้อมูล

- ยกเลิกการจอง → ใช้ปุ่มในเว็บ (นักศึกษาใช้รหัส 6 หลัก / admin ใช้หน้า admin)
- หลีกเลี่ยงการแก้/ลบแถวใน Sheet ตรงๆ ถ้าจำเป็นให้ลบ**ทั้งแถว** (อย่าล้างเฉพาะเซลล์)
- แถวข้อมูลที่เขียนโดยโค้ดเวอร์ชันเก่า (ก่อน v3) อาจมีวันที่เพี้ยน — พบแล้วให้ลบทิ้ง

## 11. ประวัติปัญหาสำคัญที่แก้แล้ว (Lessons Learned)

| ปัญหา | สาเหตุ | การแก้ |
|---|---|---|
| จองสำเร็จแต่ผังไม่แสดง + จองซ้อนได้ | Sheets แปลงข้อความวันที่เป็น Date อัตโนมัติ ทำให้เทียบสตริงไม่ตรง | `normDate_()` แปลงกลับด้วย timezone ของสเปรดชีต + บังคับคอลัมน์เป็น text + ลบแถวข้อมูลเก่าที่เพี้ยน |
